1. АНАЛІЗ ПРОБЛЕМИ ТА АНАЛОГІВ
───────────────────────────────────────────────────────────────

1.1 Аналіз проблеми

Опис проблеми:
  Сучасні організації, освітні заклади та бізнес-структури
  щоденно стикаються з необхідністю швидкого збору думок та
  прийняття колективних рішень. Голосування, опитування
  студентів, прийняття рішень у командах — всі ці ситуації
  вимагають ефективного інструменту.

Чому проблема важлива:
  1. Втрата часу на організацію — традиційні методи
     потребують 1–2 годин на одне голосування.
  2. Відсутність оперативності — затримки результатів
     призводять до помилкових рішень.
  3. Низька залученість учасників — середній response rate
     email-опитувань становить 10–30%.
  4. Відсутність real-time результатів знижує довіру
     до процесу голосування.
  5. Проблеми доступності — традиційні методи потребують
     фізичної присутності або складних налаштувань.

Як поточні підходи не задовольняють потреб:
  • Паперові голосування: тривалий підрахунок, без миттєвих
    результатів.
  • Email-опитування: низька активність, без візуалізації.
  • Google Forms: без real-time можливостей.
  • Mentimeter, Slido: дорогі ($10–50/міс), орієнтовані
    на презентації.


1.2 Аналіз аналогів

  Google Forms
    Переваги: безкоштовний, необмежена кількість відповідей.
    Недоліки: без real-time візуалізації, результати
    оновлюються лише при перезавантаженні.
    Що не вирішує: не підходить для оперативних зустрічей.

  Mentimeter
    Переваги: професійна візуалізація, real-time оновлення.
    Недоліки: $11.99/міс, у безкоштовній версії — 2 питання.
    Що не вирішує: не підходить для щоденних потреб.

  Slido
    Переваги: спеціалізація на подіях, real-time результати.
    Недоліки: від $10/event, складна настройка.
    Що не вирішує: не підходить для малих команд.

  Як нова система покращить ситуацію:
    • Миттєве створення голосування (< 30 секунд).
    • Real-time аналітика через WebSocket.
    • Мінімалістичний інтерфейс, без обов'язкової реєстрації.
    • QR-коди для швидкого доступу учасників.
    • Безкоштовна основна функціональність.


───────────────────────────────────────────────────────────────
2. БАЧЕННЯ РІШЕННЯ
───────────────────────────────────────────────────────────────

2.1 Опис призначення системи

  Система дозволяє організаторам швидко створювати голосування
  без реєстрації, а учасникам — голосувати і бачити результати
  в режимі реального часу. Адміністратор отримує повний
  контроль над усіма голосуваннями та пристроями в системі.

  Цільова аудиторія:
    • Викладачі та навчальні заклади.
    • Корпоративні організації та команди.
    • Організатори подій і конференцій.
    • Маркетологи та дослідники.


2.2 Опис бізнес-логіки

  1. Реєстрація пристрою: при першому зверненні клієнт
     отримує унікальний device fingerprint (UUID), що
     зберігається в таблиці device_fingerprints разом
     з IP-адресою та User-Agent.

  2. Створення голосування: організатор вводить заголовок,
     питання, тип (SINGLE / MULTIPLE / RATING / OPEN) та
     варіанти відповідей. Голосування прив'язується до
     fingerprint організатора. Система генерує унікальне
     посилання та QR-код.

  3. Доступ учасників: через пряме посилання або сканування
     QR-коду. Реєстрація не потрібна.

  4. Голосування: учасник обирає варіант. Сервер перевіряє
     дублювання (один fingerprint → один голос на опитування)
     та блокування пристрою. Після збереження голосу
     оновлена статистика транслюється підписникам
     через WebSocket (топік polls/{id}/results).

  5. Статистика: лічиться кількість голосів та відсоток
     за кожним варіантом; визначається лідер. Для RATING
     розраховуються середнє (mean), стандартне відхилення
     та коефіцієнт варіації.

  6. Управління: організатор може закрити голосування
     та експортувати результати у CSV або PDF. Адміністратор
     може примусово закрити, архівувати або видалити
     будь-яке голосування, заблокувати пристрій, переглядати
     логи і створювати резервні копії бази даних.

3. ФУНКЦІОНАЛЬНІСТЬ КОМПОНЕНТІВ
───────────────────────────────────────────────────────────────

3.1 Scope of Initial Release
(Обсяг першого релізу)

Перший реліз охоплює чотири компоненти: серверну частину,
веб-клієнт, мобільний клієнт та IoT-клієнт. Нижче наведено
детальний опис кожного з них на основі реалізованого коду.


  3.1.1 Серверна частина (polls-backend)

  Технологічний стек: Java 21, Spring Boot 3.5.7, PostgreSQL,
  Maven. Документація API — Springdoc OpenAPI (Swagger UI).

  REST API — реалізовані такі групи ендпоінтів:

    /api/polls
      GET    /              — перелік усіх голосувань
      GET    /{id}          — голосування за ID
      GET    /status/{s}    — фільтр за статусом (ACTIVE /
                             CLOSED / ARCHIVED)
      GET    /trending      — список за активністю (сортування
                             за кількістю голосів)
      POST   /              — створення голосування (title,
                             question, type, multipleAnswers,
                             showResults, organizerFingerprintId)
      PUT    /{id}          — оновлення заголовку, питання,
                             статусу або видимості результатів
      DELETE /{id}          — видалення голосування
      GET    /{id}/statistics — статистика: totalVotes,
                             votes/% за варіантом, лідер
      GET    /{id}/metrics  — математичні метрики розподілу:
                             mean, stdDeviation,
                             coefficientOfVariation, min, max
      GET    /{id}/qr       — QR-код та посилання на голосування
      GET    /{id}/export/csv         — CSV з результатами
      GET    /{id}/votes/export/csv   — CSV з деталями голосів
      GET    /{id}/export/pdf         — PDF-звіт (iTextPDF 7)
      GET    /export/csv              — CSV усіх голосувань

    /api/votes
      POST   /              — реєстрація голосу
                             (pollId, optionId, fingerprintId)
      GET    /              — перелік усіх голосів
      DELETE /{id}          — видалення голосу адміністратором

    /api/device-fingerprints
      POST   /              — реєстрація нового пристрою
      GET    /{id}          — інформація про пристрій
      POST   /block         — блокування пристрою
      POST   /unblock       — розблокування пристрою

    /api/admins
      CRUD-ендпоінти для управління адміністраторами.
      POST /statistics      — статистика активності за період.

    /api/admin/backups
      POST   /              — створення резервної копії БД
      GET    /              — список резервних копій
      POST   /restore       — відновлення з резервної копії
      DELETE /old           — видалення копій старіших за N днів

  Модель даних:

    Poll              — id (UUID), title, question,
                        type (SINGLE/MULTIPLE/RATING/OPEN),
                        status (ACTIVE/CLOSED/ARCHIVED),
                        multipleAnswers, showResults,
                        createdAt, closedAt, closedByAdmin,
                        organizerFingerprint, options, votes.

    PollOption        — id, poll, text, orderNum.

    Vote              — id, poll, option, fingerprint,
                        votedAt, textAnswer (для OPEN-типу).

    DeviceFingerprint — id, fingerprintHash (unique), ip (inet),
                        userAgent, createdAt, lastSeen,
                        isBlocked, blockReason,
                        blockedByAdmin, blockedAt.

    Admin             — id, name, isActive.

    AdminLog          — id, admin, action, targetType,
                        targetId, description, createdAt.
                        Фіксує дії: CLOSE_POLL, ARCHIVE_POLL,
                        DELETE_POLL, DELETE_VOTE та ін.

  Антифрод-механізм:

    Метод hasAlreadyVoted() блокує повторне голосування:
    один fingerprint може проголосувати в конкретному
    голосуванні лише один раз.

    Метод calculateAnomalyScore() обчислює співвідношення
    кількості голосів з одного пристрою до середнього
    показника по всіх пристроях. При значенні > 3.0
    isAnomalousActivity() повертає true, що сигналізує
    адміністратору про підозрілу активність.

  Real-time трансляція (WebSocket):

    При підключенні клієнт отримує sessionId та може
    підписатися на топіки:
      polls/{pollId}/results — оновлення результатів.
    Підтримуються дії: subscribe, unsubscribe, ping/pong.
    При кожному новому голосі або видаленні голосу
    WebSocketBroadcaster розсилає оновлену статистику
    усім підписникам відповідного топіку через
    ConcurrentHashMap<String, Set<WebSocketSession>>.

  QR-код:

    QrCodeService генерує посилання виду
    {baseUrl}{pollPath}/{pollId}, URL-кодує його та
    повертає URL до зображення через Google Charts API
    або qr-server.com як альтернативу.


  3.1.2 Веб-клієнт

  Технологічний стек: React + Vite, TanStack Router, Zustand,
  Tailwind CSS, Shadcn/UI, Axios, SockJS + STOMP, lucide-react.

  Функціональність для організатора:
    • Форма створення голосування (тип, питання, варіанти,
      налаштування видимості результатів).
    • Отримання унікального посилання та QR-коду для
      поширення серед учасників.
    • Трансляція результатів у реальному часі на великий
      екран під час зустрічі.
    • Таблиця з детальними результатами (кількість голосів,
      відсотки, лідер, метрики розподілу для RATING).
    • Закриття голосування та перегляд списку своїх голосувань.
    • Експорт результатів у CSV та PDF.

  Функціональність для учасника:
    • Простий інтерфейс голосування — вибір одного або
      кількох варіантів (залежно від типу).
    • Поле для текстової відповіді (OPEN-тип).
    • Real-time діаграма результатів (якщо showResults=true).
    • Повідомлення «ви вже проголосували» при спробі
      повторного голосування.

  Функціональність для адміністратора:
    • Захищена панель із JWT-аутентифікацією.
    • Список усіх голосувань з фільтрацією та сортуванням.
    • Детальна статистика по кожному голосуванню.
    • Закриття, архівування, видалення будь-якого голосування.
    • Управління заблокованими пристроями.
    • Перегляд та фільтрування логів дій.
    • Управління резервними копіями БД.
    • Загальна статистика системи за обраний період.

  Взаємодія з сервером:
    • REST API через Axios для основних операцій.
    • WebSocket (SockJS + STOMP) для підписки на
      polls/{id}/results та отримання оновленої статистики.


  3.1.3 Мобільний клієнт

  Мобільний клієнт є спрощеною версією веб-клієнта,
  оптимізованою для смартфонів та сенсорних екранів.

  Функціональність:
    • Сканування QR-коду вбудованою камерою для миттєвого
      переходу до голосування.
    • Голосування одним дотиком (SINGLE) або мультивибором
      (MULTIPLE) з адаптивним тач-інтерфейсом.
    • Відображення real-time діаграм результатів після
      голосування.
    • Push-сповіщення учасника про відкриття нового
      голосування від організатора (опціонально, через
      Firebase Cloud Messaging або аналог).
    • Відображення QR-коду на екрані для поширення
      (для організатора).
    • Локальний кеш останніх голосувань для швидкого
      повторного доступу.

  Взаємодія з сервером:
    • REST API для операцій (той самий backend).
    • WebSocket для real-time результатів.
    • Device fingerprint формується на основі унікального
      ідентифікатора мобільного пристрою.


  3.1.4 IoT-клієнт

  IoT-клієнт — це фізичний пристрій для голосування,
  призначений для конференц-залів, аудиторій та офісів.
  Він дозволяє учасникам голосувати фізичними кнопками,
  не використовуючи смартфон.

  Апаратне забезпечення:
    • Мікроконтролер (ESP32 або Raspberry Pi Zero 2W)
      з WiFi-модулем.
    • 2–6 фізичних кнопок (відповідають варіантам голосування).
    • Невеликий дисплей (OLED або e-ink, 128×64 пікс.)
      для відображення питання та поточних результатів.
    • Опціонально: вбудований QR-сканер або NFC-зчитувач
      для прив'язки до конкретного голосування.

  Принцип роботи:
    1. IoT-пристрій підключається до WiFi та реєструє
       свій device fingerprint через REST API
       (POST /api/device-fingerprints).
    2. Організатор прив'язує пристрій до активного
       голосування (передає pollId через NFC, QR або
       вручну через адмін-панель).
    3. На дисплеї відображається питання та варіанти
       відповідей, пронумеровані відповідно до кнопок.
    4. При натисканні кнопки пристрій відправляє
       POST /api/votes з pollId, optionId та своїм
       fingerprintId.
    5. Після успішного голосування дисплей показує
       поточні результати, що автоматично оновлюються
       через WebSocket-підписку на polls/{pollId}/results.
    6. Захист від повторного голосування забезпечується
       тим самим механізмом hasAlreadyVoted() серверу,
       що і для інших клієнтів.

  Взаємодія з сервером:
    • HTTP REST API (той самий backend, без змін).
    • WebSocket для відображення оновлених результатів
      на вбудованому дисплеї.

3.2 Scope of Subsequent Releases
(Інтеграція штучного інтелекту)

Другий і наступні релізи зосереджені на впровадженні ШІ
для підвищення якості системи. Нижче описано конкретні
завдання, джерела даних та технічні підходи.


  3.2.1 Застосування ШІ для покращення системи

  A. Виявлення шахрайства (Fraud Detection)

    Поточний стан: у коді вже реалізовано базовий
    anomalyScore = votesFromDevice / averageVotesPerDevice.
    Поріг > 3.0 вважається аномальним. Це проста евристика,
    яка не враховує часові паттерни, кореляції між
    пристроями та обхід через VPN.

    ШІ-покращення: навчити класифікатор (binary classification:
    «шахрай / не шахрай») на ознаках:
      — anomalyScore поточного пристрою;
      — кількість голосувань від пристрою за останню годину;
      — часовий інтервал між голосами (burst voting);
      — географічна відстань між IP-адресами одного fingerprint;
      — збіг User-Agent з відомими headless-браузерами;
      — кількість голосувань організованих з того ж IP.
    Модель може автоматично блокувати пристрої або сигналізувати
    адміністратору замість ручної перевірки.

  B. Аналіз відкритих відповідей (NLP для OPEN-типу)

    Поточний стан: поле textAnswer зберігається в таблиці
    votes, проте жодної обробки тексту немає. Організатор
    бачить лише «сирий» список відповідей.

    ШІ-покращення: автоматична кластеризація відповідей
    за темами (topic modeling), визначення тональності
    (sentiment analysis), виділення ключових слів
    та підсумок у 2–3 реченнях. Це дозволить організатору
    швидко осмислити сотні текстових відповідей.

  C. Рекомендація варіантів голосування

    Організатору пропонуються релевантні варіанти відповідей
    на основі заголовку питання та контексту (галузь, тип події).
    Це прискорює створення голосування та покращує якість
    варіантів.

  D. Передбачення фінального результату (Early Prediction)

    На основі перших 20–30% голосів ШІ-модель прогнозує
    ймовірний фінальний розподіл голосів. Це дає організатору
    розуміння тренду ще до закриття голосування.

  E. Персоналізовані рекомендації для учасників

    Система пропонує учаснику (за його fingerprint-профілем)
    активні голосування, що найімовірніше є для нього
    релевантними. Це збільшує залученість і кількість голосів.


  3.2.2 Збір даних для навчання моделей

  Наявні таблиці вже містять необхідні сирі дані:

    Таблиця votes:
      — poll_id, option_id, fingerprint_id, voted_at,
        text_answer.
      — Дає паттерни вибору варіантів та текстові відповіді.

    Таблиця device_fingerprints:
      — fingerprint_hash, ip (inet), user_agent, created_at,
        last_seen, is_blocked, block_reason.
      — Є основою для навчання fraud detection моделі.
        Позначки is_blocked=true — це готові мітки класу «шахрай».

    Таблиця polls:
      — title, question, type, created_at, closed_at, status.
      — Використовується для кластеризації тематик голосувань
        та рекомендаційної системи.

    Таблиця admin_logs:
      — action (CLOSE_POLL, DELETE_VOTE тощо), target_id,
        created_at.
      — Дії адміністраторів є додатковими мітками: якщо
        адмін видалив голоси або заблокував пристрій — це
        підтверджений факт шахрайства.

  Додатковий збір даних (потребує невеликих змін у коді):
    — Зберігати час між послідовними голосами від одного
      fingerprint (currently not stored).
    — Логувати час перебування учасника на сторінці
      голосування до моменту вибору (поведінковий патерн).
    — Зберігати результат перевірки is_anomalous у votes
      разом із фактичним рішенням адміністратора.


  3.2.3 Підхід до навчання моделей

  Фаза 1 — Використання готових API (поточні потреби):
    • Для аналізу тексту (OPEN-відповіді): Claude API або
      OpenAI GPT-4o через HTTP. Не потребує навчання,
      дає результат з першого дня.
    • Для рекомендації варіантів відповідей: LLM API з
      few-shot промптом на основі заголовку питання.
    Перевага: швидко, без потреби в датасеті.
    Недолік: вартість API-запитів, залежність від зовнішніх
    сервісів, затримка мережевого виклику.

  Фаза 2 — Дообчинання (fine-tuning) існуючої моделі:
    • Fraud detection: взяти легку модель (XGBoost або
      LightGBM) та навчити на accumulated даних з таблиць
      device_fingerprints + admin_logs за 3–6 місяців роботи.
      Мітки: is_blocked=true → клас «шахрай».
    • Для NLP-задач: fine-tune невеликої мовної моделі
      (наприклад, distilBERT або ukr-roberta) на зібраних
      текстових відповідях, якщо накопичиться > 10 000
      прикладів.
    Перевага: точність вища за евристики, низька латентність
    (інференс локально), незалежність від зовнішніх сервісів.
    Недолік: потребує достатнього обсягу даних.

  Фаза 3 — Власна модель (довгострокова перспектива):
    • Якщо система масштабується до мільйонів голосувань,
      можливо навчити власну end-to-end модель для fraud
      detection із векторними ембедингами fingerprint-профілів.
    • Розробити систему рекомендацій на основі collaborative
      filtering за патернами участі у голосуваннях.

  Рекомендований пріоритет: почати з Фази 1 (API) для
  тексту та Фази 2 (XGBoost) для fraud detection —
  це дає найшвидший результат при мінімальних ресурсах.

3.3 Limitations and Exclusions
(Обмеження та виключення)

Нижче перераховано неочевидні обмеження, що є свідомими
архітектурними рішеннями, та функціонал, явно виключений
зі скоупу.


  3.3.1 Обмеження, пов'язані з ідентифікацією

  Device fingerprint ≠ особа.
    Система ідентифікує пристрій, а не людину. Один учасник,
    який використовує різні браузери, режим «інкогніто»,
    VPN або скидає cookies, може проголосувати кілька разів.
    Це свідоме обмеження — система навмисно не вимагає
    реєстрації, щоб знизити поріг участі.

  Один fingerprint — один голос.
    У контексті MULTIPLE-типу голосування учасник може
    обрати кілька варіантів, але лише в одній сесії.
    Повторне голосування (зміна вибору) не підтримується.
    Це зафіксовано в hasAlreadyVoted(): наявність хоча б
    одного запису vote для даної пари (poll, fingerprint)
    блокує новий голос.

  IP-адреса зберігається, але не є основним ключем.
    Поле ip в device_fingerprints має тип PostgreSQL inet,
    але логіка блокування та перевірки дублювань побудована
    лише на fingerprintHash. IP використовується лише
    як допоміжна ознака для адміністратора.


  3.3.2 Обмеження реального часу

  WebSocket без STOMP-брокера.
    Реалізація використовує власний TextWebSocketHandler з
    ConcurrentHashMap для зберігання сесій. Це означає:
    — Немає горизонтального масштабування: при запуску
      двох інстансів сервера підписки не синхронізуються.
    — Немає черги повідомлень: якщо клієнт відключається
      та переєднується, він не отримує пропущені оновлення.
    — Відсутній STOMP-брокер (RabbitMQ / ActiveMQ), тому
      при великому навантаженні необхідно додати його.

  Відсутня підтримка офлайн-режиму.
    Якщо учасник втрачає з'єднання під час голосування,
    запит не повторюється автоматично. Клієнт повинен
    реалізовувати власну логіку повторних спроб.


  3.3.3 Обмеження функціоналу голосування

  OPEN-тип — лише зберігання, без обробки.
    Поле textAnswer зберігається в таблиці votes, але в
    першому релізі жодної агрегації, кластеризації або
    пошуку по відповідях немає. Організатор бачить лише
    сирий список рядків.

  Відсутнє планування голосувань.
    Немає можливості задати дату і час автоматичного
    відкриття або закриття голосування. Всі дії (відкриття,
    закриття) виконуються вручну.

  Відсутня підтримка мультимедіа в варіантах відповідей.
    PollOption містить лише текстове поле text. Зображення,
    відео або інші медіа у варіантах відповідей не підтримуються.

  Відсутня анонімна статистика для учасника в реальному часі
  при showResults=false.
    Якщо організатор вимкнув показ результатів (showResults=false),
    учасник не бачить жодних результатів під час голосування.
    Часткова анонімність (наприклад, показ загальної кількості
    голосів без розбивки) не передбачена.


  3.3.4 Явно виключений функціонал

  Реєстрація та авторизація учасників.
    Свідомо виключена для зниження порогу входу. Замість неї
    використовується device fingerprint. У результаті система
    не може гарантувати унікальність голосу на рівні особи.

  Платіжна система та монетизація.
    Перший реліз є повністю безкоштовним. Будь-яка модель
    монетизації (підписка, ліміти на кількість голосувань)
    виключена зі скоупу.

  Інтеграція з зовнішніми платформами.
    Відсутня вбудована інтеграція з Slack, Microsoft Teams,
    Zoom або LMS-системами (Moodle, Canvas). Організатор
    ділиться посиланням вручну.

  Підтримка кількох мов інтерфейсу (i18n).
    Перший реліз орієнтований на одну мову. Інтернаціоналізація
    не реалізована.

  Аналітика за географією.
    Незважаючи на те, що IP-адреса зберігається, геолокація
    та звіти за регіонами участі відсутні.

  Сповіщення електронною поштою або SMS.
    Відсутня будь-яка система нотифікацій (email/SMS/push)
    для організатора або учасників. Нагадування, підтвердження
    голосу та сповіщення про завершення виключені.

4.3 Опис програмних засобів та інструментів

  Серверна частина:
    — Java 21 + Spring Boot 3.5.7
    — Spring Web (REST API), Spring Data JPA (ORM),
      Spring WebSocket (real-time), Spring Security + JWT
    — PostgreSQL 15+ (основна СУБД)
    — iTextPDF 7.2.6 (генерація PDF)
    — Springdoc OpenAPI / Swagger UI (документація API)
    — Maven (збірка та управління залежностями)
    — Docker (контейнеризація)

  Веб-клієнт:
    — React 18 + Vite (збірка)
    — TanStack Router (маршрутизація)
    — Zustand (управління станом)
    — Tailwind CSS + Shadcn/UI (стилізація)
    — SockJS + STOMP (WebSocket)
    — Axios (HTTP-запити)
    — lucide-react (іконки)

  Мобільний клієнт:
    — React Native або Flutter (крос-платформна розробка)
    — Той самий REST API та WebSocket backend
    — Firebase Cloud Messaging (push-сповіщення)

  IoT-клієнт:
    — ESP32 або Raspberry Pi Zero 2W (мікроконтролер)

    — HTTP REST API для голосування та fingerprint-реєстрації

  Інструменти розробки:

    — Docker Compose (локальне середовище)
